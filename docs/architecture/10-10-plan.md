## Locked architecture decisions

- **Admin surface split:** Keep two apps; `apps/admin` is the platform-operator source of truth and `apps/web` remains tenant/customer (including tenant-scoped operations).
- **CI depth:** Run admin, realtime, and notifications smoke gates on every PR, accepting longer CI time for stronger safety.
- **RLS rollout:** Shipped in waves 1–5; all tenant-scoped tables covered; static check baseline cleared.
- **Boundary enforcement:** Fail CI immediately on boundary violations (no warn-only phase).
- **Scheduled work:** Platform cron-style jobs run through BullMQ repeatable dispatchers and per-org workers, not in-process `@Cron` on every API replica.

## Phase archive (completed)

The sections below record the original phase plan. All exit criteria were met by 2026-06-04 unless noted in the progress log.

### Phase 1: security and correctness baseline

**Goals:** Tenant safety non-optional in every local and CI validation flow; privileged bypass paths documented and tested.

**Exit criteria (met):**

- `pnpm validate` and PR CI fail on any tenant query policy violation.
- All bypass APIs (`withBypassRls`, impersonation paths) have dedicated tests.

### Phase 2: modularity and coupling

**Goals:** Break oversized services into bounded-context modules; reduce cross-domain imports.

**Exit criteria (met):**

- No service file over 1000 lines in API core domains.
- Ticket flows pass unchanged behavior tests before and after split.
- Domain dependency graph has no new lateral coupling violations.

### Phase 3: admin surface convergence

**Goals:** Eliminate duplicated admin responsibilities across `apps/web` and `apps/admin`.

**Exit criteria (met):**

- One documented admin architecture path ([admin-surface.md](./admin-surface.md)).
- No duplicated admin feature shipped in two apps.
- `apps/web` no longer exposes `/superadmin*` or `/dashboard/operations/{support,tenants}` (enforced by `check-web-admin-boundary.mjs`).

### Phase 4: test and operability excellence

**Goals:** Turn architecture assumptions into enforceable tests and health gates.

**Exit criteria (met):**

- CI covers core user path, admin path, and realtime path end-to-end (`tenant-auth`, `realtime-display`, `admin-auth`, `admin-dashboard`, `api-health` specs).
- Queue failure and retry behavior tested on PRs (`test-notifications-smoke` + dead-letter integration spec).
- Architecture lint checks block boundary regressions (`pnpm validate`).

Architecture size and module-boundary checks: `check:architecture:service-size`, `check:architecture:api-module-boundary`, `check:architecture:web-admin-boundary`.

### Phase 5: multi-tenant defense in depth

**Goals:** Move from partial RLS to complete tenant isolation defense.

**Exit criteria (met):**

- Tenant-scoped tables enforce org isolation at DB layer (waves 1–5; `check-tenant-isolation.mjs` baseline empty).
- Tenant isolation policy tests run in CI and pass (integration spec covers waves 1–5).
